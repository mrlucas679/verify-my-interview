# Verify My Interview Operations Runbook

Last updated: 2026-06-19

This runbook is for the live Azure Functions deployment:

- Production: `https://vmi-online-3907.azurewebsites.net`
- Canary: `https://vmi-api-3907.azurewebsites.net`
- Resource group: `rg-kkgawatlh9-6623`
- Primary package path: generated by `npm run build:functions`

## Golden Checks

Run these after every deployment or Azure configuration change:

```powershell
npm run online:smoke -- --url https://vmi-online-3907.azurewebsites.net --require-foundry --require-telemetry

$health = Invoke-RestMethod https://vmi-online-3907.azurewebsites.net/health
$health.status
$health.subsystems
```

Expected live flags:

- `foundry_agents: true`
- `shared_reports: true`
- `evidence_storage: true`
- `azure_monitor: true`
- `observability.configured: true`

`accounts` remains `false` until the owner configures a real External ID tenant
and the frontend sign-in flow is enabled.

## Deploy

Build the exact Azure Functions package used by the live app:

```powershell
npm run build:functions
```

Deploy canary first:

```powershell
az webapp deploy `
  -g rg-kkgawatlh9-6623 `
  -n vmi-api-3907 `
  --src-path dist-functions.zip `
  --type zip `
  --restart true `
  --async false

npm run online:smoke -- --url https://vmi-api-3907.azurewebsites.net --require-foundry --require-telemetry
```

Promote only after canary passes:

```powershell
az webapp deploy `
  -g rg-kkgawatlh9-6623 `
  -n vmi-online-3907 `
  --src-path dist-functions.zip `
  --type zip `
  --restart true `
  --async false
```

Then run the golden checks.

## Rollback

Keep the previous known-good zip for every production deploy. To roll back:

```powershell
az webapp deploy `
  -g rg-kkgawatlh9-6623 `
  -n vmi-online-3907 `
  --src-path <previous-known-good.zip> `
  --type zip `
  --restart true `
  --async false
```

Then run `online:smoke` and `/health`.

## Monitoring

Configure baseline Azure Monitor alerts:

```powershell
npm run azure:monitor -- -ActionEmail <operator-email>
```

Current alert set:

- `vmi-online-5xx-spike`: more than 3 server errors in 5 minutes.
- `vmi-online-latency-spike`: average response time above 30 seconds.
- `vmi-online-queue-backlog`: queued requests above 5.
- `vmi-online-high-4xx`: high client-error volume.

First response for any alert:

1. Open `/health` and confirm which subsystem changed.
2. Run `online:smoke` against production.
3. If production fails and canary is healthy, deploy the last known-good package.
4. If Foundry fails but deterministic scoring still works, keep the app online
   and mark the incident as degraded AI reasoning.
5. If upload/transcribe fail, preserve text-only verify/report flows while
   investigating OCR or Speech credentials.

## Report Moderation

Community reports are redacted before storage. A false, abusive, or duplicate
report should be removed through the admin path once auth/admin is enabled.

Until admin UI exists:

1. Capture the `reportId`.
2. Confirm the report is abusive or false-positive.
3. Remove it from Cosmos and re-run the network seed/index job if needed.
4. Keep an operator note with date, `reportId`, reason, and reviewer.

Never publish raw evidence, names, bank details, ID numbers, or private contact
details while investigating a moderation request.

## Privacy Requests

For a user deletion request after auth is enabled:

1. Verify the requester controls the account.
2. Call `DELETE /me` with that user's valid bearer token.
3. Confirm cases and evidence blobs are removed.
4. Reply with a short confirmation and do not include retained internal details.

For a company correction or objection request:

1. Ask for the exact company/domain/email/phone involved.
2. Search the de-identified report corpus for matching identifiers.
3. Remove or annotate false reports.
4. Keep scam indicators that are independently corroborated and de-identified.

## Abuse Response

If traffic spikes or bots appear:

1. Check `Http4xx`, rate-limit responses, and App Insights request volume.
2. Keep replicas low until a shared rate-limit store exists.
3. Temporarily lower `AUTH_ANON_TRIAL_MAX` only after frontend sign-in works.
4. Use Azure access restrictions only for active abuse, not as normal product
   access control.

## Cost And Quota

Check weekly:

- Function App execution count and response time.
- Foundry/model usage and throttling.
- Azure AI Search usage.
- Document Intelligence and Speech usage.
- Cosmos RU/usage.
- Storage growth in the private `evidence` container.

Set a subscription budget in Azure Cost Management once billing ownership is
confirmed. Budget alerts require the billing scope, so this is an owner action.
